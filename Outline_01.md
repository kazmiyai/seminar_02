# 最初に（5分）

## イントロダクション：なぜシステム開発はトラブルが多いのか？

### よくある失敗パターン（参加者への問いかけ）

**こんな経験、ありませんか？**

#### 1. 「言った・言わない」問題
- 「その機能、打ち合わせで話しましたよね？」
- 「いえ、聞いていません。それは契約外です」
-> ドキュメントでの証跡と承認プロセスが機能していない

#### 2. 仕様変更の想定外コスト
- 「仕様変更は軽微なので無償対応してください」
- 「いえ、これは大規模改修です。追加で〇〇万円かかります」
-> 変更の影響が可視化できていない

#### 3. プライマリ特有の「板挟み」
- **上（発注元）から**：「まだ決まりません」
- **下（開発会社）から**：「仕様が決まらないと動けない」「それは契約外です」
-> スケジュールが遅延。コストが増える。

### なぜこれらの問題が起きるのか？

**根本原因：**
1. **契約書にやるべきことや範囲が記述されていない**
   - 成果物の定義が曖昧
   - 「何をどこまでやるか」のスコープが不明確で、契約外・契約内の境界があいまい
   - トラブル時の責任範囲、費用負担、エスカレーション手順が未記載
   - 「すべてが順調に進む前提」しか想定せず、トラブル時の対応が未定義

2. **プロセスに「バッファ」と「検証ポイント」がない**
   - 開発納期 = 本番稼働日（テスト期間ゼロ）
   - 中間レビューなしで、終盤まで動くものが見えない（ブラックボックス化）

3. **プロジェクト管理の「仕組み」がない**
   - 変更管理、進捗の可視化、優先順位付けが属人的
   - 「気合と根性」で乗り切ろうとして炎上

### 本日のゴール

**「契約」と「開発のプロセス」、「プロジェクト管理」の仕組みを理解して、トラブルを未然に防ぐ**

1. **契約＝エラー処理の実装**
   - 「もし〇〇が起きたら？」という例外処理を契約条項に組み込む方法

2. **開発プロセス＝手戻りを防ぐ検証ポイント設計**
   - ダブル検収、バッファ確保、中間レビューの組み込み方

3. **プロジェクト管理＝QCDのトレードオフを可視化する仕組み**
   - 変更管理、進捗管理、リスク管理の実践手法

# 第1部：トラブルを未然に防ぐ「契約」とは（30分）

## 1-1. 契約締結までの流れ（5分）

### システム開発契約の全体プロセス

**契約締結までのステップ：**

```
【発注者（イベント主催者等）】              【受託者（開発会社）】
                                            【プライマリ（自社）】※中間に立つ場合
          ↓                                        ↓
    ┌─────────────────────────────────────────┐
    │ ステップ1：NDA（秘密保持契約）締結           │
    │   - 情報交換の前に機密保持を約束            │
    └─────────────────────────────────────────┘
          ↓                                        ↓
    ┌─────────────────────────────────────────┐
    │ ステップ2：MOU/LOI（基本合意・意向表明）     │
    │   - 大枠の合意、概算予算・スケジュール前提   │
    └─────────────────────────────────────────┘
          ↓                                        ↓
    ┌─────────────────────────────────────────┐
    │ ステップ3：RFI → RFP                        │
    │   - RFI: ベンダー情報収集                   │
    │   - RFP: 具体的な提案依頼                   │
    └─────────────────────────────────────────┘
          ↓                                        ↓
    ┌─────────────────────────────────────────┐
    │ ステップ4：見積り・提案（Proposal）         │
    │   - Phase分割or一括の判断                   │
    │   - 前提条件（Assumption）の明記            │
    └─────────────────────────────────────────┘
          ↓                                        ↓
    ┌─────────────────────────────────────────┐
    │ ステップ5：契約締結                          │
    │   - 準委任 / 請負 / フェーズ契約の選択      │
    │   - 本格的なプロジェクト開始                │
    └─────────────────────────────────────────┘
          ↓                                        ↓
        開発開始
```

#### ステップ1：NDA（秘密保持契約）の締結
- **目的**：情報交換の前に締結し、要件・仕様・見積り情報などの漏洩を防ぐ
- **対象**：イベント主催者やスポンサー情報を扱う場合は必須
- **タイミング**：初回打ち合わせ前、遅くとも詳細情報を共有する前

#### ステップ2：MOU（Memorandum of Understanding: 基本合意）またはLOI（Letter of Intent: 意向表明）
- **目的**：正式契約前に「どこまで合意しているか」を可視化する
- **内容**：
  - 役割分担の大枠
  - 概算予算・スケジュール上の前提
  - プロジェクトの方向性
- **特徴**：まだ拘束力は弱いため、変更が多い案件に向いている

#### ステップ3：要件ヒアリング・RFI/RFP提示

**3-1. RFI（Request for Information: 情報提供依頼）**
- **目的**：ベンダーの技術力や提供可能なサービスを把握する情報収集段階
- **実施内容**：
  - ベンダーの実績・技術スタック・体制の確認
  - 概算費用やスケジュールの目安を把握
  - 複数ベンダーの比較検討材料を集める
- **タイミング**：具体的な仕様が固まる前、ベンダー選定の初期段階

**3-2. RFP（Request for Proposal: 提案依頼）**
- **目的**：仕様が固まっていない段階で、方向性をすり合わせる
- **プライマリの役割**：この段階で「翻訳作業」が始まる
  - 発注元の要望をシステム要件に変換
  - 開発会社が見積もれる形に整理
  - RFIで得た情報をもとに、具体的な提案を依頼

#### ステップ4：見積り・提案（Proposal）
- **検討事項**：
  - Phase 1/2 に分けるか、一括か
  - プロジェクトの前提条件（Assumption）を明記
- **注意点**：前提条件の抜け漏れが後のトラブルの原因になる

#### ステップ5：契約締結（準委任 / 請負 / フェーズ契約）
- ここから本編（1-2以降）の契約形態の選択と設計につながる

## 1-2. 契約形態の基礎と「再委託」の構造（7分）

### 基本の2つの契約形態

#### 請負契約（成果物完成型）
- **特徴**：「完成責任」を重視。成果物の完成がゴール
- **責任**：納期までに仕様通りの成果物を完成させる義務
- **報酬**：成果物の完成・検収をもって支払い
- **リスク**：仕様変更に弱い、途中で契約解除しにくい
- **向いている場面**：要件が固まっている、予算・納期を明確にしたい

#### 準委任契約（業務遂行型）
- **特徴**：「善管注意義務」を重視。業務遂行がゴール
- **責任**：誠実に業務を遂行する義務（完成義務はない）
- **報酬**：稼働時間・期間に応じて支払い
- **リスク**：予算・納期が見えにくい、終わりが不明確になりやすい
- **向いている場面**：要件定義フェーズ、SES、保守・運用

**【重要用語】善管注意義務（善良な管理者の注意義務）とは**

準委任契約で受託者が負う責任の基準。民法第644条に規定されている。

- **定義**：
  - 「善良な管理者の注意をもって、委任事務を処理する義務」
  - その道のプロとして、通常期待される注意を払って業務を行う義務

- **具体的な意味**：
  - システム開発で言えば、「業界標準の技術力・知識を持つエンジニアとして、通常期待されるレベルの仕事をする」こと
  - 完成義務はないが、「手抜き」や「明らかな技術的ミス」は契約違反になる
  - 定期的な進捗報告・課題報告を怠ることも善管注意義務違反になる：スケジュールが遅延しているのに報告しない、不具合があるのに報告しないなど

- **請負契約との違い**：

| 項目 | 請負契約 | 準委任契約 |
|------|---------|-----------|
| 責任の対象 | **成果物の完成** | **業務遂行プロセス** |
| 未完成の場合 | 契約不履行（責任あり） | 善管注意義務を果たしていればOK |
| 判断基準 | 「できたか、できなかったか」 | 「適切に努力したか」 |

- **実務上の注意点（発注者側）**：
  - 「頑張りました」だけでは不十分。証跡（作業報告、議事録）で「何をどう頑張ったか」を確認する
  - 明らかな技術的ミス（例：セキュリティの基本対策を怠った）があれば、善管注意義務違反として追及できる
  - **報告義務違反の判断基準**：
    - 定期報告（週次・月次）がない、または大幅に遅れる
    - 重大な課題・リスクを報告せず、後から発覚する
    - 進捗の虚偽報告（「順調です」と言っていたが実は大幅遅延）
    - → これらは善管注意義務違反として契約解除や損害賠償の対象になり得る
  - 成果物が完成しなくても、契約違反とは言えないため、**進捗管理・品質チェックは発注者の責任**
  - **契約書に報告の頻度・形式を明記する**ことで、善管注意義務の基準を明確化できる

- **プライマリが注意すべきポイント**：
  - 準委任契約を下流（開発会社）と結ぶ場合、上流（発注元）への完成責任は自社が負う
  - 「開発会社が善管注意義務を果たしたが完成しなかった」場合でも、自社は発注元に対して責任を負う
  - → **下流準委任、上流請負の組み合わせは最大のリスク構造**（板挟みの典型例）

---

#### 請負契約と準委任契約のメリット・デメリット

**比較表：発注者視点**

| 項目 | 請負契約 | 準委任契約 |
|------|---------|-----------|
| **予算の明確性** | ◎ 総額固定で予算管理しやすい | △ 稼働時間次第で変動、予算超過リスク |
| **納期の確実性** | ◎ 納期コミットあり | △ 業務遂行が目的で、納期保証なし |
| **成果物の保証** | ◎ 完成義務あり、未完成は契約不履行 | × 完成義務なし、ベストエフォート |
| **仕様変更への柔軟性** | × 変更ごとに追加費用・納期延長 | ◎ 柔軟に対応可能 |
| **品質責任** | ◎ 検収基準を満たす責任あり | △ 善管注意義務のみ |
| **契約解除** | △ 違約金発生、途中解除困難 | ◎ 期間満了で終了、比較的容易 |
| **向いているケース** | 要件固定、予算・納期重視 | 要件不確定、柔軟性重視 |

**詳細解説：発注者側のメリット・デメリット**

**請負契約のメリット（発注者側）：**
1. **予算・納期が明確**
   - 総額固定で予算計画が立てやすい
   - 納期コミットがあるため、イベント開催日など動かせない期限に対応しやすい

2. **完成責任がある**
   - 仕様通りの成果物が納品されなければ契約不履行を主張できる
   - 検収基準を明確にすれば、品質を担保しやすい

3. **リスクの転嫁**
   - 開発の進捗遅延や技術的トラブルはベンダー側の責任
   - 発注者は「何を作るか」に集中できる

**請負契約のデメリット（発注者側）：**
1. **仕様変更に弱い**
   - 「ちょっとした変更」でも追加費用・納期延長の交渉が必要
   - 仕様凍結後の柔軟性がない

2. **見積もりリスクの上乗せ**
   - ベンダーはリスクを見込んで高めの見積もりを出す
   - 要件が曖昧だと、さらに割高になる

3. **途中脱出が困難**
   - 「この会社はダメだ」と気づいても、違約金を払わないと解除できない
   - ベンダーロックインのリスク

---

**準委任契約のメリット（発注者側）：**
1. **仕様変更への柔軟性**
   - 途中で方向性を変えても追加費用交渉が不要（同一稼働範囲内）
   - アジャイル開発など、段階的に仕様を詰める手法に向いている

2. **パートナーシップ型**
   - ベンダーと協力して仕様を作り上げる関係
   - 要件定義フェーズで「一緒に考える」ことができる

3. **契約解除が比較的容易**
   - 期間満了で終了、または短期の予告期間で解除可能
   - ベンダーの能力不足が判明した場合の脱出がしやすい

**準委任契約のデメリット（発注者側）：**
1. **予算・納期が見えにくい**
   - 「何人月かかるか分からない」状態が続く
   - 終わりが見えず、予算超過のリスク

2. **完成保証がない**
   - 「頑張りましたが完成しませんでした」でも契約上は問題ない
   - 成果物の品質を発注者側が厳しくチェックする必要がある

3. **発注者の管理負担が大きい**
   - スコープ管理、進捗管理を発注者が主導する必要がある
   - 「丸投げ」ができない

**【重要】準委任契約における「納期」と「完成義務」の関係**

準委任契約では、**契約書に納期を記載しても、完成義務は発生しない**（民法第656条）

- **納期の法的性格**：
  - 準委任契約の納期は「目標」または「業務遂行期間」として扱われる
  - 納期に間に合わなくても、善管注意義務を果たしていれば契約違反ではない
  - 請負契約のような「納期までに完成させる義務」とは全く異なる

- **報酬の発生タイミングの違い**：

| 項目 | 請負契約 | 準委任契約 |
|------|---------|-----------|
| 報酬発生条件 | 成果物の完成・検収時 | 稼働時間・期間に応じて |
| 未完成の場合 | 報酬請求権なし | 報酬請求権あり（善管注意義務を果たしていれば） |
| 納期遅延 | 契約不履行、損害賠償 | 善管注意義務違反がなければ賠償義務なし |

- **実務上の落とし穴**：
  - **「準委任契約」という名称でも、契約書に「成果物の完成を目的とする」と明記すると、実質的に請負契約とみなされる可能性がある**
  - 裁判で契約の実態が問われた場合、名称ではなく内容で判断される
  - 「成果完成型準委任」「準委任型請負」という中間的な契約も存在するが、法的には曖昧でリスクが高い

- **プライマリが注意すべき最大のリスク**：
  - **上流（発注元）：請負契約（完成義務あり、納期厳守）**
  - **下流（開発会社）：準委任契約（完成義務なし、納期は目標）**
  - → この組み合わせは**リスクのミスマッチ**であり、最も危険
  - 開発会社が納期に間に合わなくても、プライマリは発注元に対して完成責任を負う
  - **対策**：下流も請負契約にするか、上流も準委任契約にして契約形態を揃える

- **契約書でのリスク回避策**：
  - 準委任契約でも「中間成果物の納品期日」を設定し、進捗を可視化する
  - 「善管注意義務の具体的基準」として、週次報告・課題報告を契約書に明記する
  - 期待する成果物がある場合は、最初から請負契約を選択する

**比較表：ベンダー（受託者）視点**

| 項目 | 請負契約 | 準委任契約 |
|------|---------|-----------|
| **収益の予測** | ◎ 総額固定で売上確定 | △ 稼働時間次第で変動 |
| **リスク負担** | × 遅延・品質不良は自社負担 | ◎ 遅延リスクは限定的 |
| **仕様変更対応** | ◎ 追加費用交渉の機会 | × 同一稼働内での対応が求められる |
| **技術的チャレンジ** | △ リスクを避けたい | ◎ 試行錯誤がしやすい |
| **契約期間** | △ 完成まで拘束される | ◎ 短期・中期で契約更新 |

---

**実務上の使い分け（プライマリの視点）：**

1. **請負契約を選ぶべきケース**
   - 要件が明確に固まっている
   - 予算・納期が厳格（例：イベント開催日が確定している）
   - 発注元が「丸投げ」したい（プライマリが責任を持つ前提）

2. **準委任契約を選ぶべきケース**
   - 要件定義フェーズ（仕様が未確定）
   - 仕様変更が頻繁に発生する見込み
   - ベンダーの技術力を見極めたい（Phase 1での評価）

3. **ハイブリッド（多段階契約）がベスト**
   - **Phase 1（要件定義・設計）**: 準委任契約
   - **Phase 2（実装・テスト）**: 請負契約
   - 両方のメリットを活かし、リスクを分散

### イベント業務における3社間構造（サンドイッチ構造）

**プライマリの立ち位置：**

```
[発注元（イベント主催者）]
        ↓ (請負契約)
[自社（プライマリ）] ← 板挟みの位置
        ↓ (発注)
[開発会社（再委託先）]
```

**構造上のリスク：**
- 自社は発注元に対し「完成責任」を負っている
- しかし開発会社への発注ミスで足元をすくわれる
- **「下」の失敗を「上」にかぶる構造** = 自社がバッファとしてリスクを吸収

**リスクの具体例：**
- 開発会社が納期遅延 → 自社が発注元への遅延責任を負う
- 開発会社の品質不良 → 自社負担で修正、または損害賠償
- 再委託先の選定ミス → プロジェクト全体の失敗責任

## 1-3. リスクを回避する「契約の実務」（8分）

### 契約レビューにおける事業部と法務部の役割分担

契約書のレビューは、事業部と法務部が協力して行うことが重要です。

#### 事業部の役割（プロジェクトの実現可能性とリスク評価）

**チェック対象：「何を、いくらで、いつまでに」を中心とした実務的な内容**

1. **スコープと成果物の定義**
   - 「この仕様で本当に要件を満たせるか？」
   - 「成果物の定義が曖昧で、後で揉めないか？」

2. **納期と予算**
   - 「このスケジュールは実現可能か？」
   - 「見積もり金額は適正か？」

3. **検収・受入基準**
   - 「検収条件は具体的か？」
   - 「受入テストの方法は実務的に機能するか？」

4. **変更管理**
   - 「仕様変更が発生した場合の手続きは明確か？」
   - 「追加費用の算定方法は合理的か？」

#### 法務部の役割（法的リスクと契約条項の適法性チェック）

**チェック対象：知的財産、損害賠償、契約解除などの法的リスク全般**

1. **知的財産権の取り扱い**
   - 成果物の権利移転条項の適法性
   - 第三者のライセンス条項の確認

2. **損害賠償・免責条項**
   - 賠償責任の上限設定
   - 免責事由の妥当性

3. **契約解除・違約金条項**
   - 解除条件の適法性
   - 違約金の合理性

4. **準拠法・管轄裁判所**
   - 紛争解決手段の確認

#### 【重要】事業部が主導すべき「将来揉めそうなポイント」の洗い出し

**実は、以下のような「実務で揉めやすいポイント」は事業部の方が詳しいことが多い：**

| 項目 | 法務部の視点 | 事業部の視点 |
|------|-------------|-------------|
| **知的財産権** | 条項の適法性 | **「このライブラリはOSSか？ライセンスは？」など技術的な実態** |
| **損害賠償** | 上限額の妥当性 | **「本番障害が起きた場合の実損はいくらか？」など実務的な影響** |
| **納期遅延** | ペナルティ条項の適法性 | **「このマイルストーンが遅れるとイベント開催に間に合わない」など業務上の死活問題** |
| **検収方法** | 条項の明確性 | **「このテスト方法では不具合を検出できない」など実務上の問題** |

**望ましいアプローチ：**
- 事業部が「将来揉めそうなポイント」をリストアップ
- 法務部がそれを法的に適切な条項に落とし込む
- **事業部が主導して「例外処理」を契約に組み込む**

**具体例：**
- 事業部：「本番稼働後にバグが見つかった場合、誰がいつまでに直すのか曖昧だと揉める」
- 法務部：「では『検収後90日間は無償修正義務を負う』という条項を追加しましょう」

### 再委託特有のチェックポイント

#### チェック1：知的財産権（IP）の取り扱い
- **確認事項**：納品と同時に権利が自社に移転するか？
- **なぜ重要か**：主催者への譲渡・利用許諾に必須
- **契約条項例**：
  - 「成果物に関する一切の知的財産権は、検収完了と同時に発注者に移転する」
  - 「第三者のライブラリ使用時は、ライセンス条件を事前報告する」

#### チェック2：再委託の承諾
- **確認事項**：自社がさらに下請けを使う場合、主催者の許可が必要か
- **リスク**：無断再委託で契約違反、信頼失墜
- **契約条項例**：
  - 「事前の書面による承諾なしに再委託してはならない」
  - 「再委託先の情報（社名、責任者）を事前開示する」

#### チェック3：SLA（サービスレベル）のバック・トゥ・バック契約
- **目的**：主催者と約束した稼働率を、開発会社にも守らせる
- **Back-to-Back Agreement（背中合わせ契約）**：
  - 上流契約（主催者 ← 自社）と下流契約（自社 → 開発会社）の条件を連動させる
  - 自社がリスクを一方的に負わない仕組み
- **契約条項例**：
  - 「本番稼働後の稼働率99.9%を保証する」（上流）
  - 「受託者は稼働率99.9%を保証し、未達時は損害を補償する」（下流）

#### チェック4：損害賠償・責任上限の設定
- **確認事項**：トラブル時の損害賠償額に上限を設けるか
- **一般的な設定**：契約金額の〇〇%、または〇〇万円を上限とする
- **注意点**：上流契約と下流契約の上限額が逆転しないようにする

## 1-4. ベストプラクティス「多段階契約（フェーズ契約）」（7分）

### 一括請負の危険性

**問題点：**
- **仕様未定での総額固定はギャンブル**
  - 開発会社はリスクを見込んで高めの見積もりを出す
  - 発注側は「ぼったくり？」と感じる
- **途中脱出ができない**
  - 「この会社、ダメだ」と気づいても契約上逃げられない
  - 品質不良でも、違約金を払わないと契約解除できない

**実例：**
> 「イベント管理システム一式 800万円」という契約で、要件定義後に「実は1200万円かかります」と言われるケース。契約上は800万円で完成義務があるが、開発会社は「前提条件が違った」と主張。

### 解決策：多段階契約（フェーズ分割）

**基本構成：**

#### Phase 1：要件定義・設計（準委任契約）
- **目的**：パートナーとして仕様を固める
- **成果物**：
  - 要件定義書
  - 基本設計書
  - 画面モック
  - 詳細見積もり（Phase 2用）
- **契約形態**：準委任（〇〇人月 × 単価）
- **メリット**：
  - 開発会社の技術力・コミュニケーション力を見極められる
  - 仕様を詰めた上で、精度の高い見積もりが取れる

#### Phase 2：実装・テスト（請負契約）
- **目的**：仕様確定後、完成をコミットさせる
- **成果物**：
  - 動作するシステム
  - テスト報告書
  - 運用マニュアル
- **契約形態**：請負（成果物完成型）
- **前提条件**：Phase 1で合意した要件定義書を基準とする

**メリット：**
1. **見積もり精度の向上**：仕様が固まってから金額を確定できる
2. **Exit Strategy（脱出装置）**：Phase 1で「合わない」と判断したら、Phase 2は別の会社に発注できる
3. **リスクの分散**：大金を一度に投じるリスクを避けられる

**デメリット：**
1. **契約手続きの手間**：2回契約を結ぶ必要がある
2. **後出し見積もりの高騰リスク**：Phase 2で「やっぱり高い」と言われる可能性
   - **対策**：Phase 1の成果物に「概算見積もり範囲（±10%）」を含める

## 1-5. 【重要概念】契約書は「エラー処理」の実装である（3分）

### 契約書＝プログラムコード

**エンジニア的思考の応用：**

プログラミングで、入力値が正しい場合しか動かないコードは書かない
```
try {
    // 正常系の処理
} catch (例外) {
    // エラー時の処理
}
```

契約書も同じ。「無事に納品されました」というハッピーエンドしか書いていない契約書は、**バグだらけのプログラムと同じ**。

### 契約書に実装すべき「例外処理」

#### Catch 1：受入テストでNGが出たら？
- **契約条項例**：
  - 「検収時に不具合が発見された場合、受託者は〇〇日以内に無償で修正する」
  - 「〇回以上の修正が必要な場合、発注者は契約を解除できる」

#### Catch 2：主催者の都合でイベントが中止になったら？
- **契約条項例**：
  - 「発注者都合で契約解除する場合、既に発生した費用＋〇〇%を支払う」
  - 「不可抗力（災害等）の場合、双方協議の上、費用を按分する」

#### Catch 3：仕様変更が止まらなかったら？
- **契約条項例**：
  - 「仕様変更は変更管理表（CR）で管理し、費用・納期への影響を都度見積もる」
  - 「累積変更が契約金額の〇〇%を超える場合、契約を見直す」

#### Catch 4：納期遅延が発生したら？
- **契約条項例**：
  - 「受託者の責による遅延は、1日あたり〇〇円のペナルティ」
  - 「発注者の仕様確定遅延は、納期を延長し、追加費用を協議する」

### PMの最大の防御

**事前に「最悪の事態」を想定し、その時の挙動を契約条項にしておく**

これが書かれていないと、エラーが起きた時、プロジェクトはクラッシュし、その損害は全て自社が被ることになる。

---

**第1部のまとめ：**

✅ 契約は「信頼関係」だけでは守れない。明文化が必須。
✅ 一括請負はリスク。フェーズ分割で「見極め」と「脱出装置」を確保。
✅ 契約書は「エラー処理」。トラブル時の責任・費用・手順を事前定義。
✅ プライマリは上下の契約を連動させ、リスクを一方的に負わない設計を。
