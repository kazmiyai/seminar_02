# 2-1. 開発プロセスの全体像
## 典型的な開発の流れ

```
要件定義 → 設計 → 実装 → テスト → リリース → 保守・運用
```

### 各フェーズの概要

| 項目 | 要件定義 | 設計 | 実装 | テスト | リリース | 保守・運用 |
|---|---|---|---|---|---|---|
| **目的** | 「何を作るか」を明確にする | 「どう作るか」を決める | 実際にコードを書く | 品質を担保する | 本番環境へ移行する | 安定稼働を維持する |
| **成果物** | 要件定義書、機能一覧、画面遷移図 | 基本設計書、詳細設計書、DB設計書 | ソースコード、単体テスト結果 | テスト仕様書、テスト結果報告書、不具合管理表 | リリース手順書、運用マニュアル | 運用報告書、障害報告書 |
| **期間目安** | 全体の20-30% | 全体の20-30% | 全体の30-40% | 全体の20-30% | 1-2週間 | - |
| **発注側の役割** | 最も重要。業務要件を正確に伝える | 画面モックや設計書のレビュー・承認 | 定期的な進捗確認、デモの確認 | 受入テストの実施、不具合の優先順位判断 | リリース判断、ユーザー周知 | 問い合わせ対応、改善要望の整理 |

# 2-2. ウォーターフォール vs アジャイル

開発手法としては、大きくウォーターフォールとアジャイルがあります。
アジャイルは比較的新しい開発手法です。

## ウォーターフォール開発

**特徴：**
- 各工程を順番に完了させる
- 前工程の完了が次工程の開始条件
- 計画重視、ドキュメント重視

**メリット：**
- スケジュール・予算が見えやすい
- 大規模プロジェクトの管理がしやすい
- 契約（請負）と相性が良い

**デメリット：**
- 途中で動くものが見えない（ブラックボックス化）
- 仕様変更に弱い
- 手戻りが発生すると影響大

**向いているケース：**
- 要件が明確で変更が少ない
- 大規模システム（複数チーム連携）
- 法規制対応など、仕様変更が許されない

**V字モデル（ウォーターフォールの代表的な開発手法）：**

```
要件定義 ────────────────────────→ 受入テスト
   ↓                                    ↑
基本設計 ──────────────────→ システムテスト
   ↓                                ↑
詳細設計 ────────────→ 結合テスト
   ↓                        ↑
  実装 ──────→ 単体テスト
      ↓          ↑
       コーディング
```

- 左側（下降）：開発工程（要件→設計→実装）
- 右側（上昇）：テスト工程（単体→結合→システム→受入）
- 各工程が対応：要件定義の内容を受入テストで検証、設計内容を各レベルのテストで検証
- **ポイント**：
   - 各工程で「次工程の準備」と「対応するテスト工程の計画」を同時に行う
   - 原則は、テスト工程で問題が発生した場合、対応する設計・実装に戻る

---

## アジャイル開発

**特徴：**
- 短期間（1-4週間）のサイクルで開発とリリースを繰り返す
- 動くものを早期に確認しながら進める
- 変化への対応を重視

**メリット：**
- 早期に動くものが見られる（安心感）
- 仕様変更に柔軟に対応できる
- フィードバックを反映しやすい

**デメリット：**
- 最終的な総額・納期が見えにくい
- 「終わり」が不明確になりやすい
- 発注側の継続的な関与が必須

**向いているケース：**
- 要件が不確定、試しながら決めたい
- 小〜中規模のプロジェクト
- スタートアップや新規事業

## スクラム（Scrum）：アジャイル開発の代表的なフレームワーク

**スクラムとは：**
- アジャイル開発の具体的な実践手法の一つ
- 1-4週間の「スプリント」と呼ばれる期間で、計画→開発→レビューを繰り返す
- チーム全体で協力し、透明性・検査・適応を重視

**主要な役割：**
- **プロダクトオーナー（PO）**：何を作るかを決める（発注側の代表）
  - 優先順位を決定し、ビジネス価値を最大化する
- **スクラムマスター（SM）**：チームがスムーズに動けるよう支援する
  - 障害を取り除き、プロセスを改善する
- **開発チーム**：実際に開発を行う（少人数でチームを作る）

**スプリントの流れ：**
```
1. スプリント計画 → 2. 毎日の朝会 → 3. スプリントレビュー → 4. 振り返り
   （何を作るか）    （進捗共有）      （成果デモ）         （改善点検討）
         ↓                ↓                  ↓                  ↓
   [  1-4週間のスプリント期間で開発  ] → 動くソフトウェアをリリース
```

**主要なイベント**
1. **スプリント計画（Sprint Planning）**：今回のスプリントで何を作るか決める
2. **デイリースクラム（Daily Scrum / 朝会）**：毎日15分、進捗と課題を共有
3. **スプリントレビュー（Sprint Review）**：完成した機能をデモして、フィードバックを得る
4. **スプリント振り返り（Retrospective）**：チームのやり方を改善する

**成果物：**
- **プロダクトバックログ**：作りたい機能の一覧（優先順位付き）
- **スプリントバックログ**：今回のスプリントで作る機能の一覧
- **インクリメント**：動くソフトウェア（スプリント終了時に必ずリリース可能な状態）

**プライマリ（発注側）の関わり方：**
- プロダクトオーナーとして、または密に連携して、優先順位を常に見直す
- スプリントレビューに参加し、実際に動くものを確認してフィードバック
- 要件の変更は柔軟に可能だが、スプリント途中での変更は避ける（次のスプリントで対応）

---

## プライマリの視点：どちらを選ぶべきか
**ウォーターフォールとアジャイル**

| 項目 | ウォーターフォール | アジャイル |
|------|------------------|-----------|
| 要件の確定度 | 確定している | 不確定、変わりやすい |
| 予算の制約 | 固定したい | 柔軟に調整可能 |
| 納期の制約 | 厳格（イベント開催日など） | 柔軟 |
| 発注側の関与 | 要件定義と受入テストのみ | 継続的に必要 |
| プロジェクト規模 | 大規模 | 小〜中規模 |

**判断基準**
- 製品のライフタイム：　自社プロダクトなら継続的な改善が必要
- 予算の制約：　サービスプロバイダとして業務を受託している場合は予算の制約が大きい

**実務的な落とし穴：**
- 「アジャイルなら柔軟」と聞いて安易に選ぶと、予算超過・終わらないプロジェクトになる
- 「ウォーターフォールで完璧に」を目指すと、要件定義が終わらない
- **ハイブリッド型**：Phase 1をアジャイル的に進め、仕様確定後にウォーターフォール

# 2-3. 発注側が押さえるべきポイント
## チェックポイント1：中間レビューの設定

**問題：**
- ウォーターフォールでは「テスト工程まで動くものが見えない」
- 「期待と違った」と気づくのが遅すぎる

**対策：**
- **設計レビュー**：画面モック、DB設計を必ずレビューする
- **開発途中のデモ**：月1回程度、動くものを見せてもらう
- **契約条項に明記**：「月次で進捗報告とデモを実施する」

---

## チェックポイント2：テスト期間の確保

**問題：**
- 「開発完了 = 本番リリース日」になっている
- テストで不具合が見つかっても修正時間がない

**対策：**
- **ダブル検収の設計**：
  - 開発会社の検収（システムテスト完了）
  - 発注側の検収（受入テスト完了）
- **バッファ期間**：本番リリース日の2-4週間前を納期に設定
- **不具合修正期間**：テスト後の修正期間を明示的に確保

**契約条項例：**
```
- 開発完了期日：2025年3月15日
- 受入テスト期間：2025年3月16日〜3月29日
- 不具合修正期間：2025年3月30日〜4月10日
- 本番リリース日：2025年4月15日
```

---

## チェックポイント3：受入基準の明確化

**問題：**
- 「検収」の基準が曖昧
- 「これで完成ですか？」「まだ直すべきでは？」で揉める

**対策：**
- **受入基準（Acceptance Criteria）を事前定義**：
  - 機能要件：「〇〇ができること」を列挙
  - 性能要件：「同時100ユーザーで5秒以内にレスポンス」
  - 品質基準：「致命的バグ0件、重大バグ3件以内」

**具体例：チケット販売システムの受入基準**
```
✓ 一般ユーザーがチケットを購入できること
✓ 決済が正常に完了すること
✓ 購入履歴が確認できること
✓ 同時100ユーザーのアクセスで5秒以内にレスポンス
✓ 致命的バグ（システムダウン等）：0件
✓ 重大バグ（購入不可等）：0件
✓ 軽微なバグ（表示崩れ等）：5件以内
```

# 2-4. 最近の開発技術動向と発注側が知っておくべきこと

## なぜ技術動向を知る必要があるのか

**発注側の課題：**
- 開発会社から「最新技術で開発します」と提案される
- 「それって本当に必要？」「コストに見合う？」と判断できない
- 技術用語に振り回されて、本質的な議論ができない

**本セクションの目的：**
- 技術専門家になる必要はない
- **「提案の妥当性を判断できる」レベルの知識を持つ**
- コスト削減・リスク低減につながる選択肢を知る

---

## トレンド1：クラウド化とサーバーレス

### 従来：オンプレミス（自社サーバー）
```
初期投資：サーバー購入 500万円〜
運用コスト：保守・電気代・場所代
拡張性：サーバー増設に時間とコスト
```

### 現在の選択肢

**1. クラウド（IaaS：Infrastructure as a Service）**
- AWS EC2、Azure Virtual Machines、GCP Compute Engine
- 仮想サーバーを借りる形態
```
初期投資：ほぼゼロ（従量課金）
運用コスト：使った分だけ支払い
拡張性：数分でスケールアップ可能
```

**2. サーバーレス（FaaS：Function as a Service）**
- AWS Lambda、Azure Functions、Google Cloud Functions
- サーバー管理不要、実行した分だけ課金

**サーバーレスとは：**
- サーバーを意識せず、プログラム（関数）だけを配置する
- アクセスがあった時だけ実行され、その分だけ課金
- サーバーの運用・管理が不要

**コスト比較例：チケット販売システムの場合**

| 項目 | オンプレミス | クラウド（IaaS） | サーバーレス |
|------|-------------|----------------|-------------|
| **初期費用** | サーバー購入<br>500万円〜 | ほぼゼロ | ほぼゼロ |
| **月額費用<br>（平常時）** | 保守費 10万円 | サーバー稼働<br>15万円 | ほぼゼロ<br>（アクセス少ない） |
| **月額費用<br>（イベント期間）** | 同額 | スケールアップ<br>30万円 | アクセス数に応じて<br>5〜20万円 |
| **運用負担** | 大（自社管理） | 中（サーバー管理必要） | 小（管理不要） |

**発注側のメリット：**
- **初期コストの削減**：大規模なハードウェア投資が不要
- **柔軟性**：イベント時だけサーバーを増強、終了後は縮小
- **サーバーレスの場合**：
  - 使わない時間帯はコストゼロ
  - サーバー管理・セキュリティパッチ適用が不要
  - 自動スケーリング（アクセス急増に自動対応）

**サーバーレスが向いているケース：**
- アクセスが不定期（イベント期間中だけアクセス集中）
- 処理時間が短い（APIリクエスト、画像処理など）
- 小規模〜中規模のシステム

**サーバーレスが向かないケース：**
- 常時稼働が必要（24時間アクセスがある）
- 処理時間が長い（数分以上かかる処理）
- 既存システムとの複雑な連携

**注意点：**

1. **ランニングコストの見積もりが難しい**
   - 「使った分だけ」なので、月々の費用が読めない
   - アクセスが急増すると、予想外のコストが発生
   - **対策**：契約書に「月額上限の設定」を明記
     - 例：「月50万円を超える見込みの場合は事前通知する」

2. **コールドスタート問題（サーバーレス特有）**
   - しばらくアクセスがないと、初回起動に時間がかかる（数秒）
   - チケット販売開始の瞬間など、初回アクセスが遅くなる可能性
   - **対策**：事前ウォームアップ設定（常に一定数を起動状態に）

3. **ベンダーロックインのリスク**
   - AWS Lambda独自の機能を使うと、Azure・GCPへの移行が困難
   - **対策**：移行を見据えて、標準的な技術を使う

4. **契約書での確認事項**：
   - 想定アクセス数と、それを超えた場合のコスト試算
   - データの取り出し・移行の方法（契約終了時）
   - サーバーレスの制約（実行時間上限、メモリ上限など）

**発注側の判断基準：**

| ケース | 推奨 |
|--------|------|
| イベント型ビジネス（アクセス変動が大きい） | クラウド・サーバーレス |
| 安定稼働・予算固定したい | オンプレミスまたはクラウド（IaaS） |
| 小規模・短期間のシステム | サーバーレス |
| 大規模・長期稼働のシステム | クラウド（IaaS）またはオンプレミス |

## トレンド2：ローコード/ノーコード開発

### 定義
**ローコード/ノーコード**：プログラミングをほとんど書かずに、GUI操作でシステムを構築する手法

**主要ツール：**
- **Salesforce**：顧客管理（CRM）
- **kintone**：業務アプリ作成
- **PowerApps**（Microsoft）：社内業務アプリ
- **Bubble, Webflow**：Webアプリ作成

### メリット（発注側）
1. **開発期間の短縮**
   - 従来3ヶ月 → 1ヶ月で完成
2. **コストの削減**
   - フルスクラッチの50-70%程度
3. **内製化の可能性**
   - 非エンジニアでも簡単な改修が可能

### デメリット
1. **カスタマイズの限界**
   - 複雑な業務ロジックには向かない
   - 「ツールでできる範囲」に業務を合わせる必要がある
2. **ベンダーロックイン**
   - ツール依存が高く、他のシステムへの移行が困難
3. **ランニングコスト**
   - 月額課金が高額になる場合がある

**発注側の判断基準：**

| ケース | 推奨 |
|--------|------|
| 定型的な業務（顧客管理、申請承認など） | ローコード/ノーコード |
| 複雑な業務ロジック、大量データ処理 | フルスクラッチ開発 |
| 将来的な拡張性・柔軟性が必要 | フルスクラッチ開発 |
| 小規模・短期間で試したい | ローコード/ノーコード |

**提案評価のポイント：**
- 開発会社が「フルスクラッチ開発」を提案してきたら、**「ローコードで代替できないか？」と質問する**
- 逆に、ローコード提案を受けたら、**「将来の拡張性・移行リスク」を確認する**

## トレンド3：SaaSの活用（作らずに借りる）

### SaaS（Software as a Service）とは
**定義**：インターネット経由で利用するソフトウェアサービス

**例：**
- **Slack / Teams**：チャットツール
- **Zoom / Google Meet**：オンライン会議
- **Salesforce**：顧客管理
- **freee / マネーフォワード**：会計ソフト
- **Stripe / PayPal**：決済サービス

### 「作る」vs「借りる」の判断

**作る（フルスクラッチ開発）：**
- 初期コスト：高（500万円〜）
- ランニングコスト：低（保守費用のみ）
- カスタマイズ性：高

**借りる（SaaS）：**
- 初期コスト：低（数万円〜）
- ランニングコスト：高（月額課金）
- カスタマイズ性：低

**損益分岐点の例：**
```
SaaS：月額10万円 × 12ヶ月 × 5年 = 600万円
フルスクラッチ：初期500万円 + 保守50万円/年 × 5年 = 750万円

→ 5年以内ならSaaSが有利
→ 長期利用ならフルスクラッチも検討
```

**発注側の判断ポイント：**
1. **汎用的な機能か、独自の業務ロジックか**
   - 汎用的（顧客管理、チャットなど）→ SaaS
   - 独自性が高い（業界特有の業務）→ フルスクラッチ
2. **データの機密性**
   - 外部サービスにデータを置いても問題ないか
   - セキュリティ要件（個人情報保護法、業界規制）
3. **将来の拡張性**
   - SaaSの機能で5年後も足りるか

**実務での活用：**
- 開発会社が「決済機能を開発します」と提案 → 「Stripeで代替できないか？」
- 「顧客管理システムを作ります」 → 「Salesforceで十分では？」
- **疑問を持つことでコスト削減の余地が見える**

## 発注側が技術提案を評価する際のチェックリスト

**開発会社から技術提案を受けたら、以下を確認：**

### 1. 「最新技術」の必然性
- [ ] なぜその技術が必要なのか、ビジネス上のメリットは？
- [ ] 枯れた技術（実績のある技術）で代替できないか？
- [ ] 「最新技術」を使うことで、リスクは増えないか？

### 2. コストの妥当性
- [ ] 初期コストだけでなく、ランニングコストも試算されているか？
- [ ] SaaS・ローコードで代替した場合のコスト比較はしたか？
- [ ] 5年間のTCO（Total Cost of Ownership：総保有コスト）は？

### 3. ベンダーロックインのリスク
- [ ] この技術・サービスに依存すると、将来的に逃げられなくならないか？
- [ ] データの移行・取り出しは容易か？
- [ ] 他の開発会社に保守を引き継げるか？

### 4. 拡張性・保守性
- [ ] 将来的に機能追加しやすい設計か？
- [ ] 技術者が退職しても、引き継ぎできるか？
- [ ] ドキュメントは整備されるか？

```

```

**プライマリに求められる役割：**
1. **技術提案の翻訳**
   - 開発会社の技術提案を、発注元にビジネス言語で説明
   - 「サーバーレスアーキテクチャ」→「使った分だけ課金、イベント期間外はコストゼロ」
   - 「マイクロサービス」→「機能ごとに独立したシステムで、柔軟に拡張可能」

2. **費用対効果の評価**
   - 「最新技術」が本当に必要か、冷静に判断
   - コスト削減の選択肢（SaaS、ローコード、サーバーレス）を提示

3. **リスクの洗い出し**
   - ベンダーロックイン、拡張性、コールドスタートのリスクを評価
   - 発注元に「このリスクを取るか」判断を促す

**プライマリが陥りやすい失敗：**
- 開発会社の提案を鵜呑みにして、発注元に丸投げ
- 技術が分からないからと、判断を放棄
- 「最新技術」に飛びついて、コストが膨らむ

**対策：**
- 本セクションのチェックリストを使い、提案の妥当性を評価
- 分からないことは開発会社に質問し、納得するまで確認
- **「なぜこの技術が必要か」をビジネス言語で説明できなければ、採用しない**

# **まとめ**

✅ プライマリの役割は、技術提案を翻訳し、ビジネス判断を支援すること
✅ 開発プロセスは「要件定義」が最重要。ここでの曖昧さが後の火種になる。
✅ ウォーターフォール vs アジャイルは、要件の確定度と予算制約で判断。
✅ 中間レビュー・テスト期間・受入基準を契約時に明記することで手戻りを防ぐ。
✅ 「動くものを早めに見る」仕組みがトラブル回避の鍵。
✅ 技術動向を知ることで、開発会社の提案を適切に評価できる
✅ クラウド・サーバーレス・ローコード・SaaSは、コスト削減・期間短縮の選択肢
✅ サーバーレスは、イベント型ビジネスに適しているが、コールドスタートやコスト見積もりに注意
✅ 「最新技術」に飛びつかず、費用対効果・リスクを冷静に評価
