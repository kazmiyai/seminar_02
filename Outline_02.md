
# 第2部：「開発プロセス」の理解（15分）

## 2-1. 開発プロセスの全体像（5分）

### 典型的な開発の流れ

```
要件定義 → 設計 → 実装 → テスト → リリース → 保守・運用
```

#### 各フェーズの概要

**要件定義（Requirements Definition）**
- **目的**：「何を作るか」を明確にする
- **成果物**：要件定義書、機能一覧、画面遷移図
- **期間目安**：全体の20-30%
- **発注側の役割**：最も重要。業務要件を正確に伝える

**設計（Design）**
- **目的**：「どう作るか」を決める
- **成果物**：基本設計書、詳細設計書、DB設計書
- **期間目安**：全体の20-30%
- **発注側の役割**：画面モックや設計書のレビュー・承認

**実装（Implementation）**
- **目的**：実際にコードを書く
- **成果物**：ソースコード、単体テスト結果
- **期間目安**：全体の30-40%
- **発注側の役割**：定期的な進捗確認、デモの確認

**テスト（Testing）**
- **目的**：品質を担保する
- **成果物**：テスト仕様書、テスト結果報告書、不具合管理表
- **期間目安**：全体の20-30%
- **発注側の役割**：受入テストの実施、不具合の優先順位判断

**リリース（Release）**
- **目的**：本番環境へ移行する
- **成果物**：リリース手順書、運用マニュアル
- **期間目安**：1-2週間
- **発注側の役割**：リリース判断、ユーザー周知

**保守・運用（Maintenance & Operation）**
- **目的**：安定稼働を維持する
- **成果物**：運用報告書、障害報告書
- **発注側の役割**：問い合わせ対応、改善要望の整理

## 2-2. ウォーターフォール vs アジャイル（5分）

### ウォーターフォール開発

**特徴：**
- 各工程を順番に完了させる
- 前工程の完了が次工程の開始条件
- 計画重視、ドキュメント重視

**メリット：**
- スケジュール・予算が見えやすい
- 大規模プロジェクトの管理がしやすい
- 契約（請負）と相性が良い

**デメリット：**
- 途中で動くものが見えない（ブラックボックス化）
- 仕様変更に弱い
- 手戻りが発生すると影響大

**向いているケース：**
- 要件が明確で変更が少ない
- 大規模システム（複数チーム連携）
- 法規制対応など、仕様変更が許されない

---

### アジャイル開発

**特徴：**
- 短期間（1-4週間）のサイクルで開発とリリースを繰り返す
- 動くものを早期に確認しながら進める
- 変化への対応を重視

**メリット：**
- 早期に動くものが見られる（安心感）
- 仕様変更に柔軟に対応できる
- フィードバックを反映しやすい

**デメリット：**
- 最終的な総額・納期が見えにくい
- 「終わり」が不明確になりやすい
- 発注側の継続的な関与が必須

**向いているケース：**
- 要件が不確定、試しながら決めたい
- 小〜中規模のプロジェクト
- スタートアップや新規事業

---

### プライマリの視点：どちらを選ぶべきか

**判断基準：**

| 項目 | ウォーターフォール | アジャイル |
|------|------------------|-----------|
| 要件の確定度 | 確定している | 不確定、変わりやすい |
| 予算の制約 | 固定したい | 柔軟に調整可能 |
| 納期の制約 | 厳格（イベント開催日など） | 柔軟 |
| 発注側の関与 | 要件定義と受入テストのみ | 継続的に必要 |
| プロジェクト規模 | 大規模 | 小〜中規模 |

**実務的な落とし穴：**
- 「アジャイルなら柔軟」と聞いて安易に選ぶと、予算超過・終わらないプロジェクトになる
- 「ウォーターフォールで完璧に」を目指すと、要件定義が終わらない
- **ハイブリッド型**：Phase 1をアジャイル的に進め、仕様確定後にウォーターフォール

## 2-3. 発注側が押さえるべきポイント（5分）

### チェックポイント1：中間レビューの設定

**問題：**
- ウォーターフォールでは「テスト工程まで動くものが見えない」
- 「期待と違った」と気づくのが遅すぎる

**対策：**
- **設計レビュー**：画面モック、DB設計を必ずレビューする
- **開発途中のデモ**：月1回程度、動くものを見せてもらう
- **契約条項に明記**：「月次で進捗報告とデモを実施する」

---

### チェックポイント2：テスト期間の確保

**問題：**
- 「開発完了 = 本番リリース日」になっている
- テストで不具合が見つかっても修正時間がない

**対策：**
- **ダブル検収の設計**：
  - 開発会社の検収（システムテスト完了）
  - 発注側の検収（受入テスト完了）
- **バッファ期間**：本番リリース日の2-4週間前を納期に設定
- **不具合修正期間**：テスト後の修正期間を明示的に確保

**契約条項例：**
```
- 開発完了期日：2025年3月15日
- 受入テスト期間：2025年3月16日〜3月29日
- 不具合修正期間：2025年3月30日〜4月10日
- 本番リリース日：2025年4月15日
```

---

### チェックポイント3：受入基準の明確化

**問題：**
- 「検収」の基準が曖昧
- 「これで完成ですか？」「まだ直すべきでは？」で揉める

**対策：**
- **受入基準（Acceptance Criteria）を事前定義**：
  - 機能要件：「〇〇ができること」を列挙
  - 性能要件：「同時100ユーザーで5秒以内にレスポンス」
  - 品質基準：「致命的バグ0件、重大バグ3件以内」

**具体例：チケット販売システムの受入基準**
```
✓ 一般ユーザーがチケットを購入できること
✓ 決済が正常に完了すること
✓ 購入履歴が確認できること
✓ 同時100ユーザーのアクセスで5秒以内にレスポンス
✓ 致命的バグ（システムダウン等）：0件
✓ 重大バグ（購入不可等）：0件
✓ 軽微なバグ（表示崩れ等）：5件以内
```

---

**第2部のまとめ：**

✅ 開発プロセスは「要件定義」が最重要。ここでの曖昧さが後の火種になる。
✅ ウォーターフォール vs アジャイルは、要件の確定度と予算制約で判断。
✅ 中間レビュー・テスト期間・受入基準を契約時に明記することで手戻りを防ぐ。
✅ 「動くものを早めに見る」仕組みがトラブル回避の鍵。
