
### プロセス領域（Process Domains）

PMBOKでは、「プロセス領域」として実務的な40前後のプロセスを推奨し、ツール化。
価値提供（Value）・ガバナンス・不確実性への対応について、ウォーターフォール／アジャイル／ハイブリッドのどれでも使えるように整理されている。
必ずしもすべてのプロセスをプロジェクトに導入する必要はないが、初期段階で最低限のプロセスを導入し、問題が発生した場合に、プロセスを追加するなどの運用も可能。

#### プロセスの例
##### （１）変更管理プロセス設計（Change Control Process Design）

**変更要求の受付方法**

   - 誰が、どの形式で、どこに、変更依頼を提出するのか
   - 例：変更要求フォーム、チケット、専用ワークフロー

**影響分析の手順**
   - スケジュール、コスト、品質、リスク、リソースへの影響を評価
   - 例えば、開発・QA・インフラが分析に参加し評価

**承認フロー（意思決定者）**
   - 誰が最終判断者（A）かを明確にする
   - 変更規模によって承認者が異なる場合もありえる
   - PMOが最終判断権を持つケースも多い

**変更内容の反映（計画更新）**
   - 計画（WBS、スケジュール、予算、リリース計画）への反映手順
   - 更新後の計画を必ずステークホルダーへ共有
   - 変更理由、承認者、影響、実施日を記録

---

##### （２）コストコントロールプロセス設計（Cost Control Process Design）

**コスト実績の収集**
   - 実際に発生した工数・外注費・設備費などを収集
   - 工数管理ツール／請求情報の連携を行う

**計画（予算）との比較**
   - コスト予算（Baseline）と実績を比較
   - 乖離（差異）を定量的に把握する

**予算計画の更新**
   - 「このペースで行くと最終的にどうなるか？」を予測
   - 定期的に予算計画を更新する

**コスト超過の原因分析と対策**
   - 仕様変更によるものか、見積もりのミスか、その他の外部影響（円安など）か
   - 原因に応じて、スコープ調整・リソース再配分など対策を実施

**ステークホルダーへの報告と意思決定支援**
   - 予算の進捗状況を可視化
   - 予算追加の必要性やスコープ変更の判断材料を提供

---

##### （３）スケジュール統制プロセス設計（Schedule Control Process Design）

**進捗状況のモニタリング（実績の収集）**
   - タスクの完了率、残作業、遅延状況を確認
   - タスクボード、ガントチャート、バーンダウンを使用
   - 進捗報告は「作業時間」ではなく、未了、完了で管理

**計画との比較・差異分析**
   - 予実比較（実績 vs 計画）を行うことで、予測を行える
   - 遅延原因の特定（技術課題・見積り不足・リソース不足など）

**スケジュール予測の更新**
   - 「このペースで行くといつ終わるか？」を予測
   - 完了見込み日を更新
   - クリティカルパスへ影響が出るかをチェック
   - 遅延の波及を早期に可視化する

**必要な調整・是正処置の検討と実行**
   - 人員追加（バックフィル、外注）
   - スコープ調整（削減・後回し）
   - 優先順位の変更
   - 並行作業化（依存関係の見直し）
   - 作業の改善（無駄な作業の洗い出し、レビュー方式改善）

**ステークホルダーへの報告と意思決定支援**
   - 遅延の有無・原因・対策案をわかりやすく伝達
   - スケジュール変更が必要な場合は承認プロセスへ
   - 最新計画を共有し期待値を揃える

#### ガントチャート（Gantt Chart）

いわゆる、「線表」
Henry Laurence Ganttさんが1910年代に考案。米国のダムや高速道路といった大規模プロジェクトで使われた。
全体のスケジュールとクリティカルパスを明示し進捗管理を行う。


【ガントチャート（例）】

タスク名                  | 1週 | 2週 | 3週 | 4週 | 5週 |
-----------------------------------------------------------
A. 要件定義                | ████                        |
B. 基本設計                |       ████                 |
C. 詳細設計                |           ████             |
D. フロント実装            |               ██████       |
E. API実装                 |               ██████       |
F. 結合テスト              |                       ████ |
G. リリース準備            |                            ███|

依存関係：
B は A の完了後に開始  
C は B の完了後  
D, E は C の完了後  
F は D・E の完了後  
G は F の完了後

**メリット**
| 区分 | 内容 |
|------|-------|
| 視覚的で分かりやすい | タスクと期間が横棒で見えるため、誰でも直感的に理解できる。ステークホルダー説明にも強い。 |
| 全体計画を俯瞰できる | 各タスクの位置関係・依存関係・クリティカルタスクが一目でわかる。 |
| 進捗管理に強い | 進捗率や遅延が可視化され、早期の問題検知ができる。 |
| WBS と自然に連携できる | 作業分解（WBS）とスケジュールをそのまま結びつけられる。 |
| ステークホルダーの認識合わせが容易 | 共通のスケジュール図として合意形成に使いやすい。 |

**デメリット**
| 区分 | 内容 |
|------|-------|
| 変更管理が大変 | タスク変更が多いプロジェクトだと更新作業が重くなり、手間が増える。 |
| タスク粒度を細かくしすぎると破綻 | 1日単位など細かすぎるスケジュールは複雑化して運用が難しい。 |
| 実作業に対して過度に“予測型” | 不確実性が高いアジャイル開発だと計画がすぐ陳腐化しやすい。 |
| 人の負荷を正確に示せない | ガントは期間中心で、実際のリソース稼働や負荷状況は見えづらい。 |
| 依存関係の変化に弱い | 依存関係が変更された場合、影響範囲の手動修正が必要になる。 |

####  バーンダウンチャート（Burn Down Chart）

**アジャイル開発でよく使われる進捗可視化手法**

縦軸：残作業量（ストーリーポイント／タスク数）
横軸：時間（スプリント・日数）
残作業が理想線（予定線）どおりに減っているを管理

短いサイクルでタスクをこなしていくアジャイルに適した管理チャート

```
残作業量
  ↑
100│\
   │ \
 80│  \  理想線（計画）
   │   \
 60│    \___
   │        \___  実績線
 40│            \___
   │                \___
 20│                    \___
   │                        \___
  0└────────────────────────→ 時間
   Week1  Week2  Week3  Week4
```

**読み方：**
- 理想線より実績線が上 → 遅延している
- 理想線より実績線が下 → 順調

**メリット**
| 区分 | 内容 |
|------|-------|
| 進捗を一目で把握できる | 残作業量が直感的に見えるため、進捗遅延を瞬時に判断できる。 |
| チームのセルフマネジメントを促進 | チーム自身が残作業を見て計画調整し、主体的に改善しやすい。 |
| 毎日の変化が可視化される | 毎日の残タスクの増減がグラフになるため、流れが理解しやすい。 |
| スプリントレビューや朝会で活用しやすい | スクラムイベントでそのまま進捗の“事実”として使える。 |
| 作業の積み残しを早期検知できる | 減少傾向が弱いと早期に問題を発見し、対応に結びつけられる。 |

**デメリット**
| 区分 | 内容 |
|------|-------|
| 作業量の見積り精度に依存する | ストーリーポイントや工数見積りが不正確だとチャートが役に立たない。 |
| “量” しか見えず“質”はわからない | バグ件数、欠陥密度、品質の悪化は表現できない。 |
| タスク粒度がばらつくと正しく減らない | 大きなタスクが後半まで残るとグラフの形が歪み、正確さが下がる。 |
| 外乱の多いプロジェクトでは機能しにくい | 急な割り込み・仕様変更が多いとチャートがすぐ乱れる。 |
| チーム外の人が誤解しやすい | 線が下がらない＝チーム失敗と短絡的に評価されるリスクがある。 |

## 3-3. QCDのトレードオフ管理

### QCDの「トレードオフ」とは

**プロジェクトの鉄の三角形：**

```
        Quality（機能・品質）
           /\
          /  \
         /    \
        /      \
       /________\
   Cost        Delivery
  （コスト）   （納期）
```

**基本原則：**
- **3つ全てを同時に最大化することはできない**
- 何かを優先すれば、他が犠牲になる
- プロジェクトの成功 = 適切なトレードオフの判断

---

### PMの役割：トレードオフを可視化する

**問題：**
- 上流からの「納期も守れ、予算も守れ、機能・品質も妥協するな」という無理な要求によって、現場は疲弊
- 本来、最優先だった納期が守れない

**対策：**
- トレードオフを数値化しステークホルダーの合意を得る

**可視化の例：**

| 選択肢 | 納期 | 予算 | 機能・品質 |
|--------|------|------|----------------|
| A案：最速リリース | 4/15 | 1,200万円 | 80%（一部後回し） |
| B案：予算優先 | 5/31 | 1,000万円 | 90%（時間かけて実装） |
| C案：品質最優先 | 6/30 | 1,500万円 | 100%（全機能完璧） |

**PMの役割：**
- 納期、予算、品質（機能）それぞれの優先案を提示し、ステークホルダーの合意を得る

## 3-4. リスク管理

### リスク管理の基本

**リスク（Risk）とは：**
- 「もし〇〇が起きたら、プロジェクトに悪影響が出る」という不確実性(Uncertainty)

**リスク管理の目的：**
- リスクを事前に洗い出す
- 発生確率と影響度を評価する
- 対策を準備し、被害を最小化する

---

### リスクの洗い出し

**典型的なリスク：**

#### 技術リスク
- 「新技術を使うが、社内に経験者がいない」
- 「外部APIの仕様が不明確」
- 「既存システムとの連携が複雑」

#### スケジュールリスク
- 「要件定義が長引く可能性」
- 「テスト期間が短すぎる」
- 「主要メンバーが途中で抜ける」

#### 外部依存リスク
- 「決済代行会社の審査が遅れる」
- 「発注元の承認プロセスが遅い」
- 「サーバー調達に時間がかかる」

#### 契約・法務リスク
- 「個人情報保護法の対応が必要」
- 「第三者ライブラリのライセンス問題」

---

### リスク対応戦略

#### 1. 回避（Avoid）
- リスクそのものを排除する
- 例：新技術を使わず、枯れた技術で実装

#### 2. 軽減（Mitigate）
- リスクの発生確率や影響を減らす
- 例：テスト期間を2倍確保する、予備メンバーを確保

#### 3. 転嫁（Transfer）
- リスクを第三者に移す
- 例：外部APIのSLAを契約条項に盛り込む、保険加入

#### 4. 受容（Accept）
- リスクを認識した上で、何もしない
- 例：「軽微なバグは運用開始後に修正」

---

### リスク評価マトリクス

**発生確率 × 影響度でリスクを評価：**

```
影響度
  ↑
高│     [中]        [高]
  │   リスクB     リスクA
  │
低│     [低]        [中]
  │   リスクD     リスクC
  │
  └──────────────────→ 発生確率
      低          高
```

**例：リスク管理表**

| リスク | 内容 | 発生確率 | 影響度 | 評価 |
|--------|------|---------|--------|------|
| リスクA | 外部API仕様変更 | 高 | 高 | **最優先で対策** |
| リスクB | 主要メンバー離脱 | 低 | 高 | 対策準備 |
| リスクC | 軽微なバグ混入 | 高 | 低 | 受容可能 |
| リスクD | サーバー障害 | 低 | 低 | 無視してOK |

**運用方法：**
- 週次ミーティングでリスク管理表をレビュー
- 新たなリスクが見つかったら随時追加
- リスクが顕在化（実際に発生）したら、課題管理に移行

---

### プライマリ特有のエスカレーション

**上へのエスカレーション（発注元へ）：**
- 「開発会社から納期遅延の報告がありました」
- 「選択肢A/B/Cをご検討ください」

**下へのエスカレーション（開発会社へ）：**
- 「発注元がこの判断をしました」
- 「リカバリープランを提出してください」

**両面からのプレッシャー：**
- 上：「なんとかしろ」
- 下：「無理です」

**PMの役割：**
- 感情を排除し、**事実とデータで交渉する**
- 「無理」を「条件付きで可能」に変換する
  - 「納期厳守は無理ですが、機能を80%に絞れば可能です」

---

### ツールの活用

**プロジェクト管理ツール：**
- **Jira**：アジャイル開発向け、課題管理・スプリント管理
- **Backlog**：日本語UI、ガントチャート、wiki機能
- **Redmine**：オープンソース、カスタマイズ性高い
- **Asana / Trello**：軽量、小規模プロジェクト向け

**ドキュメント管理：**
- **Confluence**：wiki、設計書管理
- **Google Drive / SharePoint**：ファイル共有
- **Notion**：軽量wiki、議事録管理

**コミュニケーション：**
- **Slack / Microsoft Teams**：チャット
- **Zoom / Google Meet**：オンライン会議

**重要：**
- ツールは手段であり、目的ではない
- 「ツールを入れれば解決」ではなく、**運用ルールが重要**

---

**第3部のまとめ：**

✅ PMBOKは国際標準のプロジェクト管理知識体系。第7版の8つのパフォーマンスドメインは、実践的なプロジェクト管理の指針となる。
✅ QCDは全て最大化できない。トレードオフを可視化し、意思決定を促す。
✅ 進捗管理は「見える化」が全て。マイルストーン・WBS・バーンダウンで早期発見。
✅ 変更管理（CR）で「ちょっとした変更」を許さない。影響分析と承認プロセスを徹底。
✅ リスク管理は「転ばぬ先の杖」。事前に想定し、対策を準備する。
✅ エスカレーションは「丸投げ」ではなく「選択肢の提示」。事実とデータで交渉する。
✅ PMBOKの理論を、現場の実践（QCD・進捗・変更・リスク管理）に落とし込むことで、体系的なプロジェクト管理が実現できる。
