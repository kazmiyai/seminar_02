# 最初に：システム開発にありがちなトラブル

#### 1. 「言った・言わない」問題
- 「その機能、打ち合わせで話しましたよね？」
- 「いえ、聞いていません。それは契約外です」

-> ドキュメントでの証跡と承認プロセスが機能していないのは？

#### 2. 仕様変更の想定外コスト
- 「仕様変更は軽微なので無償対応してください」
- 「いえ、これは大規模改修です。追加で〇〇万円かかります」

-> 変更の影響の合意と、開発作業の可視化ができていないのでは？

#### 3. 興行元、組織委員会と開発会社の「板挟み」
- **上（発注元）から**：「要件がまだ決まりません」
- **下（開発会社）から**：「要件が決まらないので動けません」

-> 結果としてスケジュールが遅延。コストの増加につながる。

---

### トラブルの原因はここにある

**原因：**
1. **契約書にやるべきことや範囲が記述されていない**
   - 成果物の定義が曖昧
   - 「何をどこまでやるか」のスコープが不明確で、契約外・契約内の境界があいまい
   - トラブル時の責任範囲、費用負担、エスカレーション手順が未記載
   - 「すべてが順調に進む前提」しか想定せず、トラブル時の対応が未定義

2. **プロセスに「バッファ」と「検証ポイント」がない**
   - 開発納期 = 本番稼働日（テスト期間ゼロ）
   - 中間レビューなしで、終盤まで動くものが見えない（ブラックボックス化）

3. **プロジェクト管理の「仕組み」がない**
   - 変更管理、進捗の可視化、優先順位付けが属人的
   - 「気合と根性」は乗り切れない

### 本日のゴール

**「契約」と「開発のプロセス」、「プロジェクト管理」の仕組みを理解して、トラブルを未然に防ぐ**

1. **トラブルを未然に防ぐ「契約」の作成を目指す**
- 契約形態を理解する: 請負と準委任の違い
- 作業範囲（SOW）の重要性: 作業範囲・成果物・前提条件を具体的に定義する
- 契約のチェックポイント: 
   - 知的財産権、再委託承諾、SLA連動、損害賠償上限の確認
   - 「もし〇〇が起きたら？」というトラブルの想定

2. **「開発プロセス」の理解する**
- 開発プロセスの全体像: 要件定義→設計→実装→テスト→リリース→保守の流れと各フェーズの役割
- ウォーターフォールvsアジャイル: それぞれのメリデメとフィットする開発
- インフラとクラウド・サーバーレス: 最近のシステム・インフラや開発の動向
- TicketMasterや万博のシステムについての紹介
   - ダブル検収、バッファ確保、中間レビューの組み込み方

3. **プロジェクトを完遂する「プロジェクト管理」の基本を理解する**
- プロジェクト管理の基本は、QCD・スコープ・進捗・変更・リスクの5つの柱で板挟みの中バランスを取ることにある。
- PMBOK（第7版）: 国際標準の原理・原則と、実践やテーラリングなどの基本を理解する。
- QCDでの管理: 品質・コスト・納期。選択肢の数値化による意思決定の方法を知る。
- 発注側のポイント: 中間レビュー設定、テスト期間の確保、受入基準の明確化など
- 進捗・変更・リスク管理: マイルストーン/WBS管理。変更管理（CR）プロセス管理、リスクの管理を実践できるようになる。
